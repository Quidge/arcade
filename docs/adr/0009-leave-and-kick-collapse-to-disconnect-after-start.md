# Leave and Kick collapse to Disconnect after the GameSession starts

Pre-Start (in the lobby), `Leave` removes a Player's seat from the GameSession and `Kick` removes the target's seat, both freeing the display name and the slot toward the 8-Player cap. This is what ADR 0008 records and what #7 shipped.

Post-Start (after the first Round has begun), `Leave` and `Kick` no longer remove the seat. They collapse into the same operational transition as `Disconnect`: the seat persists, the per-Player `Connected` flag flips to false, server-held Drafts and Chain anchoring are preserved, Ghost fills the Player's missing Entries at Round-end, and the Host drives the Player's Chain at reveal time per ADR 0004. The verb the Player or Host pressed is *not* preserved in domain state — once the GameSession has started, all three causes produce the same Disconnected seat.

This holds because:

- A GameSession's Player count is locked at Start (the N→N Chains invariant in `CONTEXT.md`). Freeing a name or a slot mid-game has nothing to attach to — no one can Join an in-progress GameSession.
- ADR 0004 names "left, kicked, disconnected" as the three causes of an absent starter at reveal time. Reading those as prose enumeration of *scenarios* rather than three distinct state-machine states — which is how the ADR is most naturally read — collapses them into one operational state: "the starter is not currently connected." This ADR makes that reading explicit.
- The Host-migration timing distinction between voluntary Leave (immediate) and Disconnect (15s grace) — recorded in #7 — falls out automatically pre-Start (the seat is destroyed, so there is nothing to wait on) and is uniform post-Start (15s grace applies in both cases, because the seat persists and the server cannot distinguish a voluntary departure from a connection drop without an additional flag). The accepted cost is a 15-second migration wait in the post-Start voluntary-Leave case, which is not a meaningful UX regression for a co-located party game.

Kick mid-game is therefore "soft": the kicked Player can technically Reconnect under the same display name, since the seat persists and ADR 0008's replace-always rule lets any WebSocket upgrade typing a held name take the seat. This is accepted per ADR 0003's trust model (co-located trusted friends; impersonation is already an accepted risk). The Host's escape valve for a determined griefer is ending the GameSession entirely — out of scope here, but the eventual answer.

## Considered Options

- **Reject Leave and Kick entirely once the GameSession has started** (rejected) — would force a phase guard at the WebSocket command layer and hide the affordances in the template after Start. Hostile UX: a Player whose phone is dying or who needs to step away has no clean way to signal "stop waiting on me," they just close the tab. Also requires meaningful rework of the shipped `gamesession.Leave` (it would need a new `ErrCannotLeaveMidGame` error and a phase check).
- **Preserve a `Left` flag on the post-Start seat that blocks Reconnect for the rest of the GameSession** (rejected) — would give Kick teeth (the kicked Player could not just retype their name and be back) and would preserve a UI distinction between "left voluntarily" and "(disconnected)". Rejected because the trust model already accepts impersonation as an in-character risk, the social signal of a Kick is usually enough in a co-located room, and the alternative escape valve (Host ends the GameSession) handles the rare hard-griefer case. The simplicity gain of "no `Left` state" outweighs the lost Kick eviction.
- **Phase-uniform Leave that destroys the seat in-game too** (rejected) — would honor #7's prose ("Leave removes the target's seat") literally, but breaks ADR 0004's premise that a Chain whose starter has left still travels and is driven at reveal by the Host. Cannot honor both ADR 0004 and #7's literal text simultaneously. We honor ADR 0004 because the alternative — handling chain deletion mid-game, the reveal renumbering, and the "what happens to the Entries the absent Player already authored on other Chains" question — is meaningfully more design work for no user benefit.
- **An `intent: voluntary` annotation on the seat that preserves the Leave-vs-Disconnect distinction for UI/migration-timing purposes only** (rejected) — earlier sketch in this slice's grilling session. The flag's only operational role would be skipping the 15s Host grace timer on post-Start voluntary Leave. The cost (an extra field threaded through wire format and event handling) outweighed the benefit (a 15s wait nobody minds). Dropped in favor of "post-Start migration is uniform; pre-Start migration timing falls out of the seat-destruction rule."

## Consequences

- `gamesession.Leave` (currently destroys the seat unconditionally per #7) needs a phase check. In the lobby, behavior is unchanged. Post-Start, `Leave` flips `Connected` to false and emits `PlayerDisconnected` — the same transition as the `Disconnect` verb. `PlayerLeft` events fire only in the lobby case.
- The `kick` command in the web layer, similarly, dispatches to seat removal pre-Start and to a `Disconnect` of the target post-Start. The close-frame reason on the kicked WebSocket may still read "kicked" so the targeted Player sees the social signal; the underlying state is just Disconnected.
- ADR 0004's "left, kicked, disconnected" framing is now explicitly read as prose enumeration of scenarios, not as a typology of three distinct state-machine states. No edit to the prose is strictly required; ADR 0004 gets a one-line forward-reference to this ADR for future readers.
- ADR 0008's intro sentence ("The seat is removed only by Leave (Host kick, voluntary 'leave game,' or GameSession ending)") is true pre-Start; post-Start the seat is removed only by GameSession ending. Gets a one-line clarification.
- `CONTEXT.md`'s Leave verb entry gets a clause noting that Leave behaves phase-dependently. The Host abilities list keeps "kicking other Players" as a power, but its mid-game effect is force-disconnect rather than seat removal.
- #7's text states "If the Host Leaves voluntarily, the auto-migrate rule fires immediately (no grace timer)." This holds pre-Start (the seat is destroyed). Post-Start, the rule becomes "Host departure of any kind starts a 15s grace timer." #7 did not actually exercise the post-Start case — mid-game Host management was not built — so this is a forward decision about not-yet-shipped behavior, not a regression on shipped code.
- Round 0 (issue #8) inherits one absent-starter mechanism instead of three. The Ghost fill / Host-driven-reveal pattern from ADR 0003 + ADR 0004 handles all of voluntary Leave, Kick, and Disconnect uniformly.
