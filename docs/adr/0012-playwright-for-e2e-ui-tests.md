# Playwright for end-to-end UI tests

The project takes on `@playwright/test` as the runner for the e2e tier defined in ADR 0006, replacing that tier's "prose contracts verified manually" stopgap with executable, CI-gateable browser-driven tests. This supersedes ADR 0006's rejection of Playwright; that rejection was conditioned on a small visual surface that did not justify a Node toolchain in dev and CI, and explicitly noted that the calculus would change "when the UI grows past easy human eyeballing." It has.

Concretely: the UI now spans the lobby, host controls, two distinct Round flows (caption entry and drawing), per-Chain Reveal with starter-driven pacing, and end-game tear-down — each driven through a live WebSocket with state transitions that are difficult to eyeball at PR review without missing regressions. The protocol layer is covered by the integration tier under `tests/integration/`, but the rendered DOM transitions, the WebSocket-driven roster updates, and the multi-tab interactions are not — and the gap is widening with each slice that ships.

`@playwright/test` is the chosen runner because it is the boring, well-trodden answer for this job: cross-browser (Chromium, Firefox, WebKit) from a single API, multi-context support that maps cleanly onto Scribble's "Alice and Bob in two tabs" testing pattern, auto-wait semantics that eliminate the test-flake category most often caused by hand-rolled `sleep` calls, and built-in tracing for debugging failures in CI. It is widely adopted, well-documented, and the canonical choice for this tier of testing in 2026.

## Considered Options

- **Playwright (chosen).** Standard, deterministic, CI-gateable, requires a Node toolchain. Specs written in TypeScript live alongside the existing Go test tiers under `tests/`. Runs unattended; produces HTML reports and traces on failure.
- **A "scenario-testing skill" driven by `chrome-devtools-mcp`** (rejected for CI; kept available for exploratory work) — the path ADR 0006 originally pointed at. The MCP browser tools are real and useful for ad-hoc walkthrough and persona-flavored sanity checks during development, but they require Claude in the loop and are not (today) deterministic enough to gate a CI pipeline on. The two approaches are not mutually exclusive: Playwright covers the regression-protection floor; MCP-driven exploration covers what humans-with-a-browser would otherwise do by hand. This ADR commits only to Playwright; the MCP route remains available without further ceremony.
- **Cypress** (rejected) — viable alternative, similar capability surface. Loses on cross-browser breadth (WebKit support is bolted on rather than native), and the API ergonomics for multi-tab / multi-context testing are weaker than Playwright's `browser.newContext()`. Playwright's docs coverage on context7 is also stronger, which matters for agents working in this codebase.
- **Stay on prose contracts + manual PR-review eyeballing** (rejected) — the option ADR 0006 chose. Revisited because the UI surface has grown past the point where the cost of automation is dominated by the cost of missed regressions. ADR 0006 itself flagged this revisit point.

## Consequences

- **New runtime in the dev and CI loop.** Node and pnpm are now required to run the full check matrix. The repo gains `package.json`, `pnpm-lock.yaml`, `.nvmrc`, and `playwright.config.ts` at the root, and a `node_modules/` directory that is gitignored. ADR 0001 (no JavaScript build step) stands — there is still no Node toolchain in the *production* path; the runtime is dev-and-test only.
- **Browser binaries are large** (~500 MB across Chromium, Firefox, WebKit) but cached at `~/Library/Caches/ms-playwright/` outside the repo. CI will need to install them via `pnpm exec playwright install --with-deps` before the test stage.
- **The third tier of ADR 0006 is renamed.** What was `tests/scenario/` becomes `tests/e2e/`, and the tier's description shifts from "prose contracts pending an executor" to "Playwright specs (TypeScript)." ADR 0006 is amended accordingly. The original prose contracts (`0001-...md`, `0002-...md`) have been ported to `.spec.ts` files and removed; new e2e coverage lands as Playwright specs from the outset.
- **`justfile` gains a `test-e2e` recipe.** `test-all` currently does not run e2e by default — e2e is slower, requires the binary to be built and a Node toolchain present, and is invoked as a separate stage in CI rather than alongside `go test`. Revisit if/when run-times allow folding it in.
- **`docs/stories/` remains parked.** A persona-flavored narrative tier was considered alongside this change; the duplication concern with prose-style e2e tests was a major reason that idea felt soft. With e2e now being code rather than prose, the question of whether stories add value as a separate narrative tier becomes cleaner to answer — but it is not answered by this ADR.
- **Reversibility:** this is reversible-with-effort. If Playwright proves wrong for the project, the specs can be deleted, the package.json and lockfile removed, and the prose contracts re-emphasized as the manual-verification artifact. The cost is the spec work already invested; there is no protocol or schema lock-in.
