# E2E admission: cheapest tier wins, specs map to user journeys

A test case belongs in the e2e tier (`tests/e2e/`, Playwright per ADR 0012) only if no lower tier can give equal confidence. "Lower tier" means unit (ADR 0006 tier 1) or whole-app integration (ADR 0006 tier 2). The decision question for every proposed e2e case is one line: *can a unit or integration test prove this behavior at meaningfully lower cost?* If yes, push down.

Operationally this rule has two corollaries:

1. **Each e2e spec maps to one user journey** — a coherent end-to-end capability the product promises to a user, observed through the actual browser. New cases belong in an existing spec only if they are meaningful variations on that spec's journey; a distinct journey warrants its own spec; anything else goes down a tier.
2. **The lobby-journey and full-game-journey are the canonical two for the foreseeable slices.** Concretely: `0001` is the lobby journey at N=10 (Join, watch the roster fill, see the cap-rejection copy on an 11th attempt, Host clicks Start, all clients transition — stop). `0002` is the full game journey at N=3 (Start through Round N-1 through per-Chain Reveal through End game). Together they exercise the two product flows whose failure modes are only observable through a real browser at user perspective.

The rule is named "cheapest tier wins" rather than framed as a set of admission criteria deliberately. Earlier framings of this question for the project — most recently a three-criterion discriminator built around DOM, cross-tab WebSocket fan-out, and browser event semantics — described what e2e *can* uniquely test without forcing the question of whether a given case *needs* the unique capability. The cost-asymmetry framing fixes that: it makes the contributor argue, per case, that the cheaper tier would miss the failure mode.

The underlying argument is the canonical testing pyramid (Cohn, *Succeeding with Agile*, 2009) as developed by Fowler ("TestPyramid", bliki, 2012) and revisited by Vocke ("The Practical Test Pyramid", martinfowler.com, 2018) and Dodds (the "testing trophy", 2018). The pyramid is fundamentally a cost optimization rather than a purpose taxonomy: higher-tier tests are slower (milliseconds → seconds → minutes), more brittle (a button-text change breaks a UI test while leaving underlying logic untouched), give poorer failure localization ("something is wrong" vs. "this function returned X instead of Y"), and give better integration coverage. The pyramid prescription that follows is: get the most coverage from the cheapest tests; use the expensive tier only where cheap tests cannot go. Vocke's load-bearing rule, "don't test the UI to test business logic," is the same rule stated negatively. Dodds's trophy revision, which argues for a fat integration middle and a *very* thin e2e top in typical web apps, is the same rule taken to its conclusion for SaaS-style projects at the scribble scale.

## Considered Options

- **No explicit admission rule (status quo before this ADR)** (rejected). Lets the e2e tier accumulate cases by default. ADR 0006 declared the three tiers and noted minimum-bar e2e coverage for new visual surface, but left the *maximum* unspoken. In practice, "is this important?" is always answerable yes, so without a discriminator the top of the pyramid inflates. The brittleness and maintenance cost are paid one PR at a time and only become visible after the suite is already heavy.
- **Generous purpose framing: any core user functionality deserves an e2e** (rejected). Appealing because it tracks how product owners think about features. The problem is that "does the functionality work" is what *every* tier proves at its scope — a unit test proves it at the function level, an integration test at the service level. As an e2e-specific purpose statement it does not discriminate, and so it justifies adding e2e cases for every feature. Fowler's "TestPyramid" addresses this directly: tiers are defined by scope and cost, not by which behaviors are "important enough" to test.
- **Three-criterion discriminator: e2e iff DOM rendering / cross-tab WS / real browser events** (rejected). The criteria are correct as far as they go, but they describe what e2e *can* uniquely cover without forcing the contributor to argue that the case needs unique coverage. A cap-rejection case meets the DOM criterion in the weak sense (it renders a string), so the discriminator admits it, even though an integration test with a small DOM-snapshot assertion gives the same confidence at a fraction of the cost. Cost-asymmetry as the primary lens cuts through these borderline calls more reliably than enumerated criteria.
- **Trophy-extreme: one e2e spec ever, the full user journey** (rejected at this slice's maturity). Defensible for very small or very early-stage apps. Rejected because the lobby flow at N=10 has a distinct failure mode (rendering and behavior at the seat cap) that the full-game journey at N=3 does not exercise. Two specs is genuinely warranted; one would be under-covering. The trophy heuristic ("very few e2e specs") is honored without taking it to the limit.

## Consequences

- **0001 tightens.** Its current scope (lobby behaviors plus reconnect plus exact share-link assertions) is reduced to the lobby user journey: Join through Start with cap-rejection along the way, at N=10. Behaviors that were in 0001 only because Playwright existed (Reconnect, exact text of the share link) move to integration tests with light DOM-snapshot assertions where the rendered HTML matters.
- **0002 stays substantive.** Full game playthrough at N=3, Start through Reveal through End. This is the canonical "user journey" e2e the trophy model would have you write, and remains the primary regression-protection artifact for the rendered game flow.
- **New e2e cases require explicit user-journey justification.** A PR adding a case to an existing spec or a new spec entirely must argue *which* journey the case advances and *why* a lower tier would miss the failure mode. Reviewers can push back on PRs that add e2e cases without that argument; the rule is intentionally written to be reviewable.
- **Inline spec headers shrink to a pointer.** Each spec's header comment becomes a one-line scope statement plus "see ADR 0013 for admission rules" rather than an inline restatement. The rules live here so they cannot drift between specs.
- **ADR 0006 is amended.** A forward-pointer at the bottom of 0006 notes that this ADR adds e2e admission discipline. 0006's substantive content (the three-tier declaration) is unchanged — this ADR is additive, not a reversal. The amendment follows the repo's established pattern (cf. ADR 0001's "Scope: production only" addendum referencing ADR 0012).
- **ADR 0012 is amended** with a parallel forward-pointer. 0012 chose Playwright as the runner and addressed mechanics; this ADR narrows what gets written *in* Playwright specs without changing the runner choice. A reader landing in 0012 from a search for "Playwright" reaches this ADR directly rather than via 0006.
- **No tooling enforcement.** The rule is a review discipline, not a lint check. A future slice could add CI signal (e.g., fail the build if `tests/e2e/*.spec.ts` files grow above a soft byte threshold) but introducing tooling now would be premature — the rule needs to land and prove itself before being mechanized.
